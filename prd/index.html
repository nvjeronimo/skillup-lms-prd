1Product Overview#

1.1 What This Is

SkillUp LMS is a corporate learning platform where employees enroll in courses and programs to develop professional skills. This PRD covers the learner-facing experience for browsing, tracking, and completing learning content.

The platform supports three content structures (Course Program TechMaster) across three delivery methods (Flexible Learning Online Live Classes Flexible + Online Live Classes). Learners move through enrollment states from Not Started through In Progress to Completed, with live session attendance, assessments, projects, and certifications along the way.

1.2 Problem Statement

The current platform was built as a technical LMS focused on administration, cohorts, and certification logic. Learners experience it as an admin panel, not a learning tool.

  1. No clear answer to "What should I do next?" across any page
  2. Progress shown as raw percentages and tables (back-office language, not motivation)
  3. Certificate eligibility displayed as rejection ("You are not eligible") with no path forward
  4. Course structure is a flat accordion list with no visual hierarchy
  5. Dates page shows admin timestamps, not learning pace
  6. Zero emotional feedback. The platform never acknowledges effort or progress
  7. Navigation exposes internal system logic ("View in Studio", "Stack") that means nothing to learners
  8. No distinction between programmes and courses for new users

1.3 Design Goals

  1. Make the next action impossible to miss
  2. Show progress as motivation, not raw data
  3. Reduce anxiety around certification
  4. Keep the interface simple, calm, and outcome-focused
  5. Optimize for desktop first (1440px), mobile planned for Phase 2
  6. Dense professional aesthetic: high information density, monochromatic base, color reserved for status and actions

1.4 Target User

Adult learner, employed, enrolled in at least one programme. Goal-oriented and time-constrained. Wants clarity, reassurance, and momentum. Not interested in gamified or playful UI. Expects a professional tool that respects their time.

1.5 Visual Direction

Bloomberg Terminal meets Linear meets Jira. Dense professional. Every pixel earns its place.

  • High information density with compact vertical rhythm
  • Clear hierarchy through weight and size contrast, not decoration
  • Monochromatic base (grays, near-blacks, whites) with color reserved for status and actions
  • Tables and lists over cards where possible
  • Subtle borders over shadows. One level of elevation max
  • Data-forward: KPIs, percentages, counts, and dates prominent and scannable

2.3 Delivery Methods (Mode of Delivery)

Core principle: Course structure and delivery method are separate concerns. A course is a package of content (title, modules, lessons, quizzes, projects). VILT is not a course type. It is a mode of delivery. The same course structure applies regardless of how it is delivered. Never create separate VILT and self-paced instances of the same course.

How delivery mode works: The mode of delivery is tagged at any level of the content hierarchy: lesson, module, or course. The tag determines whether content is consumed independently by the learner or delivered live by an instructor. Modes bubble up: if all lessons in a module are VILT, the module displays as VILT. If some are self-paced and some VILT, the module displays as Blended.

MethodBadgeWhat It MeansLearner Experience
Self-PacedFlexible LearningNo instructor involvement. Learner consumes content on their own schedule.No deadlines. No calendar entries. No timeline pressure. Can take as long as needed.
VILT (Instructor-Led)Online Live ClassesHeavily instructor-driven delivery. Still includes quizzes, projects, reading material. Think university: lectures + homework.Schedule-dependent. Start/end dates visible. Calendar entries with date, time, presenter. Recordings after session. Nudges for upcoming sessions.
BlendedFlexible + Online Live ClassesMix of self-paced and live sessions within the same course or module.Start/end dates visible. Self-paced content available anytime. Live sessions on schedule. Calendar shows only the VILT portions.

Tagging rules:

  • Delivery mode is tagged on the admin side during course creation, at lesson, module, or course level.
  • If VILT or Blended is selected, the system requires: session date, time, presenter name, and meeting link.
  • Mode bubbles up automatically. If all lessons in a module are VILT, the module shows VILT. Mixed = Blended. All self-paced = Self-Paced.
  • There is no such thing as "pure VILT" with zero supporting content. Even fully instructor-led courses have quizzes, assessments, and reading material.

No prerequisite locking for VILT: A learner who hasn't finished modules 1 and 2 can still attend the live session for module 3. The system sends a nudge ("Your session on X is in 3 days. You haven't completed the prerequisite modules.") but does not block attendance.

Badge spec: Montserrat SemiBold 10px, cornerRadius 100px (pill), padding 8px horizontal / 3px vertical. Implemented as a reusable CourseTypeBadge component set with 3 variants.

8Live Sessions (VILT) Architecture#

Source: Discovery meeting April 10, 2026 (Harpreet Kaur, Govind Sharma, Navdeep Malhotra, Rupali Amit Mahale, Nelson Jeronimo). Full architecture diagram: VILT-Architecture-Flowchart.html

8.1 How Delivery Mode Is Applied

The admin creates a course with its full structure (modules, lessons, quizzes, projects). Separately, they tag the mode of delivery at any level:

  1. At lesson level: Lesson 1 is self-paced, Lesson 2 is an industry expert session (VILT). This is the most granular tagging.
  2. At module level: The entire module is delivered by an instructor. All lessons within inherit the VILT tag.
  3. At course level: The entire course is instructor-led. All modules and lessons inherit.

When a lesson or module is tagged as VILT, the system requires: session date, time, presenter name, and meeting link. Without these, the VILT tag cannot be saved.

8.2 Bubble-Up Logic

The delivery mode displayed at parent levels is derived automatically from children:

ScenarioLessonsModule Displays As
All self-pacedL1: SP, L2: SP, L3: SPFlexible Learning
MixedL1: SP, L2: SP, L3: VILT, L4: VILTFlexible + Online Live Classes
All instructor-ledL1: VILT, L2: VILT, L3: VILTOnline Live Classes

The same logic applies at course level (derived from modules) and program level (derived from courses).

8.3 How the Learner Sees It

In the Course Structure (Linear View)

  • All modules and lessons are listed in order, exactly as self-paced courses are today.
  • Each item shows a visual indicator (badge, icon, or color) for its delivery mode: self-paced vs. live.
  • VILT items show the session date and time inline next to the lesson/module title.
  • Completed VILT sessions show a "Done" status. If a recording is available, show a "Watch Recording" link.

Calendar Tab (Course Level)

  • A tab alongside the course structure (next to Modules, About, etc.).
  • Shows only the live sessions for this specific course.
  • Each entry: session topic, date, time, presenter, meeting link.
  • Correlates directly with the VILT items in the course structure view.

Global Calendar

  • Accessible from the learner dashboard or My Learning.
  • Aggregates all live sessions across all enrolled courses and programs.
  • The learner sees their full schedule at a glance: "Tuesday: AI session. Thursday: Cybersecurity session."

Nudges and Reminders

  • Upcoming session reminder: "Your session on [topic] is in 3 days."
  • Prerequisite nudge: "You haven't completed Modules 1 and 2. Your upcoming session covers these topics. Please review before attending."
  • No blocking: the learner can still attend the session even if behind on self-paced work.

8.5 Business Rules

  • One course, one instance. Never create separate VILT and self-paced versions of the same course. This was a concrete problem: learners were going to two different places for the same content.
  • "Pure VILT" still has supporting content. Even a fully instructor-led course includes quizzes, projects, and reading material. That doesn't make it "blended." Think university lectures: the professor teaches, but students still have textbooks and assignments.
  • Self-paced = no timeline pressure. No deadlines, no schedule. The learner can take as long as they need.
  • VILT = time-bound. Any VILT component adds start/end dates, schedule pressure, and calendar entries to the learner view.
  • Content types at lesson level. Approximately 40 content/interaction types are available (video, reading, quiz, lab, drag-and-drop, jigsaw, etc.). The delivery mode is independent of the content type.

8.6 TechMaster AIML Example (Real Case)

TechMaster AIML has 16 self-paced courses. For the first course (AI Fundamentals, 1-month duration):

  • Self-paced modules available Monday through Friday: 4 modules with lessons, reading material, labs.
  • 8 VILT sessions on weekends: each summarizes self-paced content + covers advanced topics not in the LMS.
  • This is a Blended course. The module structure stays the same. The VILT sessions appear in the course structure with dates/times and in the course-level calendar tab.
  • The learner sees: "Module 1 (Self-Paced) > Module 2 (Self-Paced) > Session 1: AI Lifecycle Overview (Sat, Apr 12, 10:00 AM) > Module 3 (Self-Paced) > Session 2: Ethical AI Governance (Sat, Apr 19, 10:00 AM)"

12Edge Cases and Business Rules#

  • Multiple live sessions at same time: No conflict detection. Need indicator + priority suggestion.
  • Mandatory vs Optional: No visual differentiation. Need compliance badge, deadline indicator.
  • Course archived while enrolled: No design for access, grace period, frozen content notice.
  • Instructor changed mid-course: No notification design.
  • Timezone: "(GMT)" suffix on session times. Full handling deferred.
  • Accessibility: Calendar screen reader, keyboard nav for cards, focus management in filters, red/green blindness (~8% male users).
  • Syllabus button destination: Undefined (modal, PDF, or new page).

15Cohort Management Requirements#

Source: FRD-Cohort Management (V2 Draft, Mar 13, 2026). Authors: Harpreet Kaur, Rupali, Govind, Simran, Roop.

Problem

Currently manages cohorts by duplicating the entire program instance per cohort. Creates content duplication, complex updates, inconsistent experience, high admin effort. Goal: single reusable program structure with multiple concurrent cohorts.

Learner Workflow (9 steps)
  1. Enrollment: System assigns learner to cohort based on availability/preference
  2. Cohort-specific access: Modules, sessions, discussions, projects all scoped to cohort
  3. Release logic: Courses unlock based on cohort schedule (e.g., 3 days before Session 1)
  4. Live session attendance: Scheduled session, link provided, attendance recorded
  5. Self-paced learning: Access released content, complete quizzes, track progress
  6. Project/lab submission: File upload, link, or form. Feedback and resubmission.
  7. Progress tracking: Three dimensions (academic, cohort, completion eligibility)
  8. Completion: Evaluated on scores, project completion, attendance
  9. Certification: Certificate of Completion OR Participation. Admin/instructor releases.
Multi-Dimensional Progress Model

Academic Progress: % courses completed, quiz/assessment completion, scores.

Cohort Progress: % sessions attended, % labs/projects submitted, milestones.

Completion Eligibility: % mandatory criteria fulfilled for certification.

Configurable weighted model. Only released components count. At-risk conditions: low progress by mid-program, missed consecutive sessions, overdue submissions.

Admin Workflow: 18-Step Cohort Lifecycle
  1. Create program (once)
  2. Define completion criteria
  3. Create cohort
  4. Add learners
  5. Assign instructor
  6. Create session calendar
  7. Map sessions to modules
  8. Add session links
  9. Configure release rules
  10. Assign labs/projects
  11. Configure notifications
  12. Start cohort
  13. Monitor progress
  14. Track attendance
  15. Review submissions
  16. Mark completion
  17. Release certificates
  18. Archive cohort
